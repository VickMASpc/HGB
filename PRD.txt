PRD — Python 2D Point-and-Click Engine + Visual Editor

Project codename: TBD
Primary stack: Python, pygame-ce, Dear PyGui
Product type: Engine + visual editor toolkit
Target platform: Desktop only
Primary user: Solo developer
Development mode: AI-agent-assisted implementation, suitable for Codex-style task execution
Workflow basis: This PRD proceeds after Step 0 and Step 0b, following the supplied consulting workflow: challenge assumptions, create strategic planning, generate backlog, define implementation details, and close with a synergy report.

1. Executive Summary

Build a 2D Python point-and-click adventure engine and visual editor inspired by classic children’s adventure games such as Putt-Putt.

The project must allow a solo developer to create a small adventure game without manually editing JSON. The first production target is a 3-scene mini adventure where the user can visually create scenes, set hotspots, define NPC subtitle lines, configure scene transitions, set character movement paths, and playtest the result through a pygame-ce runtime.

The editor is not secondary. The engine and editor are equally important. The runtime must be well-structured and independent from the editor, while the v1 editor should remain simple, functional, and focused.

2.1 Simulated Expert Roles

Role 2 — Editor Tools Engineer

Role 3 — Adventure Game Systems Designer

2.2 Critical Risks in the Current Idea
Risk 1 — “No walkable areas” conflicts with “character walks to exit point”

You want no general walkable-area system, but you also want transitions to occur only after the character walks to an exit point. That is workable, but it means v1 needs an explicit movement-path model.

The correct v1 interpretation is:

The user draws or configures simple movement paths and destination points per hotspot/exit. The character does not freely pathfind across the scene.

This avoids full pathfinding while still supporting the desired adventure-game feel.

Risk 2 — Building the editor before the data model stabilizes will cause rework

The editor must manipulate scenes, hotspots, NPCs, subtitles, transitions, paths, layers, and assets. If the JSON schema is unstable, every editor panel will break repeatedly.

Therefore, the project must be schema-first:

Define project schema.
Define scene schema.
Define interaction/action schema.
Define path schema.
Build runtime loader.
Then build editor panels around those structures.
Risk 3 — Dear PyGui and pygame-ce are separate UI/rendering worlds

The runtime uses pygame-ce; the editor uses Dear PyGui. They should not share rendering code. They should share data models, serialization, and validation only.

Correct separation:

shared/
  schemas
  model classes
  validators
  serialization

runtime/
  pygame-ce renderer
  game loop
  scene manager
  input system

editor/
  Dear PyGui UI
  scene canvas
  property panels
  asset management
Risk 4 — “Visual/no-code” can become too broad

A full no-code editor can become a project larger than the engine. v1 must avoid general-purpose visual programming.

The v1 action system should be deliberately small:

on_click hotspot:
  - say subtitle
  - move player to point/path
  - change scene
  - run dialogue sequence

No arbitrary Python scripting in v1.

3. Product Vision
3.1 Product Goal

Create a desktop toolchain for building simple 2D point-and-click adventure games in Python.

The product consists of:

Runtime engine
Runs playable games using pygame-ce.
Visual editor
Lets the developer build scenes, place objects, configure interactions, set movement paths, edit NPC subtitles, validate the project, and playtest.
Shared project format
Stores all game data in portable project folders using JSON files.
3.2 v1 Product Definition

v1 is complete when the developer can create the following entirely through the visual editor:

A 3-scene mini adventure:
- Player can move around each scene.
- User-created hotspots trigger subtitles.
- One hotspot transitions to another scene.
- One NPC displays multiple subtitle lines.
- All scenes and interactions are created through the visual editor.

9. Core Data Model
9.1 Entities

Required v1 entities:

Project
Scene
Layer
AssetReference
SpawnPoint
Player
Hotspot
Exit
NPC
DialogueLine
Action
MovementPath
ValidationIssue
9.2 Coordinate System

Use a simple 2D pixel coordinate system:

Origin: top-left
X: increases right
Y: increases downward
Units: pixels

All editor and runtime coordinates must use the same coordinate space.

9.3 Hotspots

v1 supports rectangular hotspots only.

A hotspot must have:

- id
- name
- rect: [x, y, width, height]
- enabled
- action list

Polygon hotspots are deferred.

9.4 Movement Paths

Because v1 has no walkable areas and no pathfinding, movement must be defined by explicit point paths.

A path is:

[
  [120, 380],
  [250, 380],
  [400, 360]
]

Rules:

- Paths are manually drawn or edited in the visual editor.
- The player follows each point in order.
- Scene transition exits require a walk_path.
- Interaction can optionally require movement before action execution.
- No obstacle avoidance in v1.
9.5 Dialogue

v1 dialogue is linear subtitle-only dialogue.

Rules:

- No branching.
- No voice audio.
- No portraits unless trivial to add.
- Dialogue blocks interactions while visible.
- Dialogue can advance by click or timed delay.
10. Interaction Model
10.1 Standard Input

Runtime input:

Left click empty scene:
  - optional movement if a target path or point exists
  - otherwise no effect

Left click hotspot:
  - run hotspot action list

Left click NPC:
  - run NPC dialogue action

Left click exit:
  - move player along configured exit path
  - after reaching final path point, change scene

Escape:
  - close runtime or open simple debug/menu placeholder

F1:
  - toggle debug overlay
10.2 Interaction Blocking

While subtitles are visible:

- Hotspot clicks are ignored.
- Scene transitions are ignored.
- Player movement input is ignored.
- Click advances dialogue if configured.
11. Action System
11.1 Design Rule

Use a visual predefined trigger/action system. Do not implement Python scripting in v1.

11.2 Supported v1 Actions
say

Displays one subtitle line.

{
  "type": "say",
  "speaker": "Player",
  "text": "It's locked."
}
dialogue

Runs linear NPC lines.

{
  "type": "dialogue",
  "npc": "dog"
}
move_player

Moves player along a configured path.

{
  "type": "move_player",
  "path": [
    [200, 380],
    [300, 370]
  ]
}
change_scene

Changes active scene.

{
  "type": "change_scene",
  "scene": "clubhouse",
  "spawn": "entrance"
}
sequence

Runs multiple actions in order.

{
  "type": "sequence",
  "actions": [
    {
      "type": "say",
      "speaker": "Player",
      "text": "Let's go."
    },
    {
      "type": "move_player",
      "path": [[400, 380], [700, 380]]
    },
    {
      "type": "change_scene",
      "scene": "clubhouse",
      "spawn": "entrance"
    }
  ]
}
12. Runtime Flow
12.1 Startup
1. Load project folder.
2. Parse game.json.
3. Validate project.
4. Load start scene.
5. Load assets for start scene.
6. Spawn player.
7. Enter main loop.
12.2 Main Loop
1. Poll pygame events.
2. Convert input to runtime commands.
3. Update active actions.
4. Update player movement.
5. Update dialogue state.
6. Resolve scene transition if pending.
7. Render scene.
8. Render player/NPCs.
9. Render subtitles.
10. Render debug overlay if enabled.
12.3 Scene Transition Flow
1. Player clicks exit hotspot.
2. Runtime blocks additional interactions.
3. Runtime moves player along configured exit walk_path.
4. Runtime verifies final path point reached.
5. Runtime loads target scene.
6. Runtime places player at target spawn.
7. Runtime unblocks interactions.

14. Validation Requirements

The editor and runtime must share the same validation logic.

14.1 Required Validation Rules

Validation must detect:

- Missing game.json
- Invalid schema version
- Missing start scene
- Scene file listed in game.json does not exist
- Scene id mismatch
- Missing background image
- Missing player sprite
- Missing NPC sprite
- Missing target scene
- Missing target spawn
- Exit without walk_path
- Scene without any spawn point
- Duplicate ids within a scene
- Invalid rectangle dimensions
- Invalid path point format
- Missing dialogue NPC reference
- Missing action target
- Broken asset path
14.2 Validation Severity

Use three severities:

ERROR:
  Prevents running.

WARNING:
  Allows running but indicates likely problem.

INFO:
  Helpful non-blocking notice.
14.3 Runtime Behavior on Invalid Data
Missing optional sprite:
  Show placeholder rectangle.

Missing required background:
  Fail scene load with clear error.

Missing target scene:
  Block transition and log error.

Scene with no spawn:
  Fail validation before runtime.

Overlapping hotspots:
  Warning only in v1.

19. AI Agent Implementation Instructions
19.1 General Rules for Agents

AI agents working on this project must obey these rules:

- Do not build features outside v1 scope.
- Do not introduce inventory, audio, save/load, branching dialogue, or pathfinding.
- Do not couple editor code to pygame runtime internals.
- Do not store absolute asset paths in project files.
- Do not skip validation.
- Do not implement Python scripting for game logic in v1.
- Do not optimize prematurely.
- Prefer small, testable modules.
- Update tests when changing schema behavior.
19.2 Preferred Coding Style
- Python 3.11+
- Type hints required.
- Small classes with clear responsibilities.
- Dataclasses or Pydantic models acceptable.
- Pure shared models must be independent of GUI/rendering libraries.
- Runtime systems should be deterministic where practical.
- Avoid global mutable state except for constants/config.
19.3 Error Handling Standard

All loader and validator errors must include:

- severity
- code
- human-readable message
- file path if applicable
- object id if applicable

Example:

{
  "severity": "ERROR",
  "code": "MISSING_TARGET_SCENE",
  "message": "Exit 'to_clubhouse' references missing scene 'clubhouse'.",
  "file": "scenes/town_square.json",
  "object_id": "to_clubhouse"
}

23. Definition of Done
23.1 v1 Definition of Done

v1 is done only when:

- Runtime runs independently from editor.
- Editor creates a portable project folder.
- Project data is saved as relative-path JSON.
- User can create scenes visually.
- User can place rectangular hotspots visually.
- User can resize hotspots visually.
- User can create NPCs visually.
- User can edit NPC subtitle lines visually.
- User can create scene exits visually.
- User can draw exit movement paths visually.
- User can validate the project.
- User can playtest current scene.
- User can run full game.
- User can create the 3-scene mini adventure without manually editing JSON.
- Automated tests pass.
24. Roadmap After v1

After v1, consider features in this order:

1. Better player animation states.
2. Foreground occlusion layers.
3. Inventory.
4. Save/load gameplay state.
5. Audio.
6. Branching dialogue.
7. Cutscene action sequences.
8. Packaging with PyInstaller.
9. Localization.
10. Polygon hotspots.
11. Walkable areas/pathfinding.
12. Plugin support.

Do not start these until v1 is accepted.

25. Implementation Prompt for Codex-Style Agent

Use this prompt when delegating implementation:

You are implementing a Python 2D point-and-click adventure engine and visual editor.

Build the project according to the PRD.

Hard constraints:
- Runtime uses pygame-ce.
- Editor uses Dear PyGui.
- Runtime and editor are equally important.
- Runtime must run independently from editor.
- Shared models must not import pygame-ce or Dear PyGui.
- Project files must be portable JSON with relative asset paths.
- v1 supports rectangular hotspots only.
- v1 has no inventory, no audio, no save/load, no branching dialogue, no pathfinding, no polygon hotspots, no Python scripting.
- Character movement uses explicit point paths drawn in the editor.
- Scene exits trigger only after the player walks to the exit path endpoint.
- Dialogue is subtitle-only and linear.
- Interactions are configured through visual trigger/action lists.
- The editor must support drag/drop object placement, grid snapping, layers, resizable hotspots, NPC subtitle editing, project validation, playtest current scene, and run full game.

Prioritize:
1. Shared schema and validation.
2. Runtime loading and rendering.
3. Runtime interactions.
4. Editor project loading/saving.
5. Visual scene editing.
6. Action/dialogue editing.
7. Playtest workflow.
8. 3-scene acceptance project.

Do not implement features outside v1 scope.
Keep modules small and testable.
Add or update tests for schema, validation, loading, and action execution.
26. Feasibility Analysis

This project is feasible for a solo developer because the scope is intentionally narrow:

- 2D only
- desktop only
- pygame-ce runtime
- Dear PyGui editor
- no inventory
- no audio
- no branching dialogue
- no pathfinding
- rectangular hotspots only

The main complexity is not rendering. The main complexity is the editor-data-runtime loop.

The project should succeed if implementation starts from schema and validation rather than UI.

27. Identified Residual Risks
1. Dear PyGui canvas interaction may become awkward for precise scene editing.
2. Runtime/editor coordinate conversion can cause subtle bugs.
3. Movement paths may feel unnatural without walkable areas.
4. Asset copying and relative path management can become messy if not centralized.
5. Visual action editing can expand beyond v1 scope.
6. Autosave may corrupt user expectations if it overwrites primary files, so autosave must stay separate.
7. Lack of pathfinding means the user must author believable movement manually.
